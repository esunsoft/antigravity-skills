# 使用本地隔离工作区

## 概述
本地工作区用于在不依赖 git 分支或 git worktree 的前提下，为任务创建独立的工作目录。

**核心原则：** 先选择足够轻量、但能避免污染源目录的隔离方式，再在动手前完成最小基线验证。

**开始时说明：** "我正在使用 using-local-workspaces skill 来准备本地隔离工作区。"

## 何时使用
以下场景适合使用本 skill：
- 需要把生成文件与源目录隔离开
- 想先做实验，不直接改原始文件
- 正在做批处理，输出应该写到独立目录
- 需要本地沙箱，但当前项目不使用 git
- 希望降低源文件与生成产物混杂的风险

以下场景不适合使用本 skill：
- 任务只是在原地安全地修改一两个文件
- 用户明确要求直接在当前目录中修改
- 隔离带来的成本明显高于任务本身，且几乎没有污染风险

## 目录选择流程
按以下优先级处理：

### 1. 检查现有目录
优先沿用已有约定，顺序如下：
1. `.local-workspaces\`
2. `workspaces\`

如果两者都存在，优先使用 `.local-workspaces\`。

### 2. 检查 CLAUDE.md
如果项目文档中已经指定了工作区或沙箱目录偏好，直接使用。

### 3. 询问用户
如果既没有现有目录，也没有项目文档约定，则询问用户：

```text
当前没有发现本地工作区目录偏好。你希望我把隔离工作区建在哪里？

1. .local-workspaces\ （项目内隐藏目录）
2. workspaces\ （项目内可见目录）
3. %USERPROFILE%\.claude\workspaces\<project-name>\ （用户级目录）

你偏好哪一种？
```

## 隔离策略
选择**满足需求的最轻量隔离方式**。

| 场景 | 策略 |
|------|------|
| 只需要隔离生成结果 | 输出隔离 |
| 只需要少量文件或子目录 | 局部复制 |
| 需要在某个子项目上做大范围或高风险试验 | 完整工作区复制 |

### 输出隔离
源文件保持原位，只为报告、导出结果、转换产物等建立独立输出目录。

适用于：
- SQL 审查输出
- 文档或报告生成
- 数据或代码产物导出

### 局部复制
只把本任务需要的文件或子目录复制到隔离工作区。

适用于：
- 只需在少量文件上测试改动
- 源目录很大，但任务范围很小
- 完整复制成本过高

### 完整工作区复制
把相关项目或子项目整体复制到隔离工作区。

适用于：
- 预计会修改很多文件
- 试验可能破坏原目录稳定性
- 需要一个可反复试错的完整沙箱

## 准备流程
### 1. 确定工作区名称
使用简短、面向任务的名称，例如：
- `sql-audit-run`
- `dw-fix-pass-1`
- `report-template-test`
- `pricing-export`

### 2. 创建工作区目录
在选定的工作区根目录下创建目标目录。

### 3. 按隔离策略准备文件
- 输出隔离：只创建输出目录
- 局部复制：只复制需要的文件和目录
- 完整工作区复制：复制相关工作树或子项目

### 4. 执行最小基线验证
在正式修改前，先验证隔离工作区可用。

可接受的基线检查示例：
- 必要输入文件存在
- 输出目录可写
- 脚本或工具能启动且不会立即失败
- 当前任务依赖的配置文件存在
- 如果任务有测试或验证命令，运行与项目匹配的最小验证

**如果基线失败：** 报告失败情况，并询问是先修复还是带着风险继续。

**如果基线通过：** 报告工作区已就绪。

### 5. 报告工作区状态
使用简洁格式，例如：

```text
本地工作区已就绪：<full-path>
隔离策略：<输出隔离 / 局部复制 / 完整工作区复制>
基线检查：通过
可以开始处理：<task-name>
```

## 快速参考
| 情况 | 动作 |
|------|------|
| `.local-workspaces\` 存在 | 使用它 |
| `workspaces\` 存在 | 使用它 |
| 两者都存在 | 使用 `.local-workspaces\` |
| 两者都不存在 | 先看 CLAUDE.md，再问用户 |
| 只需要隔离输出 | 用输出隔离 |
| 只需要少量文件 | 用局部复制 |
| 预计有大量高风险修改 | 用完整工作区复制 |
| 基线验证失败 | 先报告，再询问是否继续 |

## 常见错误
### 默认做完整复制
- **问题：** 浪费时间和磁盘空间
- **修正：** 除非确实需要大范围编辑，否则优先考虑输出隔离或局部复制

### 跳过基线验证
- **问题：** 后续失败容易被误判为本次修改引入
- **修正：** 开始前总是先验证最小输入、输出和进入条件

### 把生成文件混进源目录
- **问题：** 清理困难，也不利于判断真正改了什么
- **修正：** 只要不需要改源码，就优先使用独立输出目录

### 假设所有任务都需要隔离
- **问题：** 给简单安全的修改平添流程成本
- **修正：** 只有当隔离确实能降低风险或减少污染时才使用本 skill

## 红线
**绝不要：**
- 假设 git 必然可用或必须使用
- 引入基于分支的操作说明

## 与其他工具的关系

### vs. Git Worktree
- **Git Worktree**：适用于 git 仓库，基于分支隔离，共享 .git 目录
- **本 Skill**：适用于任何目录（包括非 git 项目），基于文件系统复制隔离

**何时使用本 Skill**：
- 项目不使用 git
- 需要隔离输出文件而非代码分支
- 只需要临时沙箱，不需要版本控制

**何时使用 Git Worktree**：
- 项目使用 git 且需要分支隔离
- 需要在多个分支间切换工作
- 需要保留 git 历史和提交能力
- 对所有任务都默认使用完整复制
- 在目录偏好不明确时跳过询问
- 未做基线验证就宣称工作区已就绪

**始终要：**
- 优先选择最轻量、但足够安全的隔离方式
- 遵循目录优先级：现有约定 > CLAUDE.md > 询问用户
- 修改前先做基线验证
- 清楚报告工作区路径和隔离策略
